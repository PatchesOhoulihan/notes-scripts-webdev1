Model View Controlller

➤ wie eine Anwendung aufgebaut ist.
➤ was MVC bedeutet und warum man sich danach richten sollte.

1. MVC

Der norwegische Informatiker Trygve Reenskaug hat sich darüber bereits 1979 Gedanken gemacht und ein allgemeines Konzept vorgestellt, das sich als De-facto-Standard für den Grobentwurf von Anwendungen durchgesetzt hat. Das Muster wird Model-View-Controller (kurz MVC) genannt siehe T. M. H. R EENSKAUG (1979).

Wie der Name schon andeutet, wird eine Anwendung gemäß MVC in drei Einheiten strukturiert:

  • das Datenmodell (engl. model)
  • die Präsentation (engl. view) und
  • die Steuerung (engl. controller).

Das Datenmodell

Der Zweck jeder Anwendung ist schlussendlich die Verwaltung von Daten — egal ob es um eine Komponente in der CAD-Software, einen Text im Editor oder eine virtuelle Welt im Videospiel geht.

Zur Verwaltung im Rechner muss die Komponente, der Text oder die virtuelle Welt abstrahiert und in Form von Daten erfasst sein. Diesen Teil bezeichnet man als das Datenmodell.

Example:

class Model { constructor() { this.spielerX = 100; this.spielerY = 150; this.richtung = 'oben'; } };

Präsentation und Steuerung

Die Präsentation kümmert sich um die Darstellung des Datenmodells und der Elemente der Benutzerschnittstelle.

Eine komplexe Anwendungen besteht aus mehreren Präsentationen, die verschiedene Aspekte des Datenmodells darstellen und dem Benutzer verschiedene Arten der Interaktion ermöglichen.

Wenn sich das Datenmodell ändert, muss die Präsentation entsprechend reagieren.

Wichtig: Die Präsentation reagiert nicht selbst auf Benutzereingaben! Dafür ist die dritte Komponente zuständig, nämlich die Steuerung.

Die Steuerung wertet Benutzereingaben aus und reagiert entsprechend. Sie kann das Datenmodell ändern, die Präsentation aktualisieren oder wechseln.

Den Code, der das Datenmodell ändert, nennt man auch Geschäftslogik. Je nach Ansatz kann die Geschäftslogik Teil des Datenmodells oder Teil der Steuerung sein.

2. MVC im Web

Beachten Sie, dass MVC ein Muster für einen Grobentwurf ist. Die Details werden sicherlich je nach verwendeter Plattform, Programmiersprache und Framework anders umgesetzt werden.

Plattformen

Unter Plattform versteht man die Umgebung, in der eine Anwendung läuft.

Bei einer Webanwendung ist die Plattform auf der Clientseite eigentlich schon vorgegeben: Es ist der Webbrowser (HTML, CSS, JavaScript).

Interessanter ist die Serverseite. Dort gibt es eine überraschend große Auswahl an sol- chen Plattformen. Sie unterscheiden sich im Betriebssystem, in der Programmiersprache, der Webserver- und der Datenbanksoftware. Die Diskussion, welche denn nun die beste Plattform ist, füllt seit Jahren die Internetforen.

Frameworks

Die meisten Entwickler erfinden das Rad nicht gerne ständig neu, daher verwenden sie Frameworks: Das sind »Rahmen«, bestehend aus Codebibliotheken und Richtlinien, wie diese zu benutzen sind.

Frameworks gibt es sowohl für serverseitige als auch für clientseitige Entwicklung — und zwar wie Sand am Meer. Ein beliebtes Serverframework für Node.js ist z. B. Express. Für clientseitige Entwicklung seien jQuery und AngularJS erwähnt.

Das gilt insbesondere für Webanwendungen — Reenskaug hat MVC ja bereits 1979 eingeführt, lange vor der Erfindung des World Wide Web!

Einige Frameworks folgen neueren Varianten von MVC wie z. B. Model-View-ViewModel (MVVM), die besser auf moderne Plattformen übertragbar sind. Das populäre Framework AngularJS sagt über sich selbst, dass es das MVW-Muster einsetzt (Model-View-Whatever) und meint damit »whatever works for you« (I. M INAR (2009)).

Wie wird MVC im Web umgesetzt.

Zunächst einmal muss festgehalten werden, dass der Webbrowser selbst eine Anwendung ist. Für den Webbrowser ist der HTML-Code der angezeigten Webseite das Datenmodell, das in einem Fenster präsentiert wird. Die Steuerungselemente des Webbrowsers (etwa Vor- und Zurück-Button, URL-Leiste, Scrollbar usw.) beeinflussen diese Präsentation.

Die Einheiten nach MVC müssen auf Webbrowser und Webserver verteilt werden. Die Kommunikation zwischen den beiden Seiten erfolgt mittels HTTP (siehe Hinweis-Box »Das HTTP- Protokoll«).

Bei einer klassischen Webanwendung kümmert sich der Webbrowser lediglich um die Präsentation. Die Steuerung (und das Datenmodell) laufen im Webserver. Jeder Klick muss also erst einmal per HTTP zur Steuerung an den Webserver geschickt werden. Daraufhin ändert die Steuerung das Datenmodell (falls nötig) und aktualisiert die Präsentation, indem sie eine neue HTML-Seite als Antwort zum Webbrowser schickt.

Wie bereits in Abschnitt 1.1 angedeutet, heißt die Lösung Single-Page-Application: Dabei wird die Steuerung oder zumindest relevante Teile davon in den Webbrowser verlagert; eventu- ell, je nach Natur der Anwendung, sogar das Datenmodell, oder Teile davon. Ein Klick wird dann lokal (im Webbrowser) ausgewertet, und die betroffenen HTML-Elemente werden sofort aktualisiert. Falls eine Folgekommunikation mit dem Server nötig ist (etwa um das Datenmodell auf dem Server anzugleichen)

Das HTTP-Protokoll

HTTP (Hypertext Transfer Protocol) ist ein Standard, der beschreibt, wie ein Webbrowser oder ähnliche Software (Client) mit einem Webserver (Server) kommuniziert.

Die Kommunikationseinheiten, die Client und Server austauschen, werden als Nachrichten bezeichnet. Davon gibt es zwei Arten: Der Client sendet Anfragen (engl. request) zum Server und dieser sendet Antworten (engl. response) zurück.

Jede Nachricht besteht aus einigen Textzeilen (engl. header), gefolgt von einer leeren Zeile und (falls vorgesehen) einem Datenteil.

Hier ist die HTTP-Anfrage, die Ihr Webbrowser an den Server de.wikipedia.org schickt, wenn Sie die URL:

http://de.wikipedia.org/wiki/Hypertext_Transfer_Protocol aufrufen:

GET /wiki/Hypertext_Transfer_Protocol HTTP/1.1
Host: de.wikipedia.org

(Die dritte Zeile ist leer, sie signalisiert das Ende des Headers. Datenteil ist keiner vorhanden.)

Der Wikipedia-Server sendet daraufhin folgende HTTP-Antwort an den Webbrowser zurück:

HTTP/1.1 200 OK
Server: Apache
Content-Type: text/html; charset=UTF-8


[... Datenteil mit dem HTML der Seite ...]

Neben der sogenannten GET-Anfrage in diesem Beispiel kennt HTTP noch andere Arten von Anfragen. Wichtig ist die POST-Anfrage, mit der ein Client Daten zum Server übertra- gen kann — zum Beispiel die Daten, die über ein HTML-Formular abgeschickt werden, und noch einige andere mehr.
